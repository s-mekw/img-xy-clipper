## 1. 正常系テストケース

### TC-001: PNG画像の正常読み込み

- **テスト名**: PNG画像を読み込み正しいメタデータとBase64データが返る
  - **何をテストするか**: PNG形式の画像ファイルをload_imageに渡した際に、正しいImageMetadata構造体が返却されること
  - **期待される動作**: ファイルをデコード→寸法取得→フォーマット判定("png")→Base64エンコード→ImageMetadata返却
- **入力値**: テスト用PNGファイルのパス（`tests/fixtures/sample.png`、100x50px程度の小さい画像）
  - **入力データの意味**: 最も基本的な対応形式であるPNG画像の正常パス
- **期待される結果**:
  - `Ok(ImageMetadata)` が返る
  - `width` が画像の実際の幅と一致（例: 100）
  - `height` が画像の実際の高さと一致（例: 50）
  - `format` が `"png"` である
  - `base64` が空文字列でない
  - **期待結果の理由**: 要件定義REQ-001の受け入れ基準「PNG画像を読み込み表示できる」に直接対応
- **テストの目的**: load_image関数の基本的な正常系動作確認
  - **確認ポイント**: 全フィールド（base64, width, height, format）が正しく設定されること
- 🔵 要件定義書・タスク定義・既存実装から直接導出

- **期待される結果**:
  - `Ok(ImageMetadata)` が返る
  - `width` が画像の実際の幅と一致（例: 80）

### TC-003: Base64データが有効でデコード可能であること（PNG）

- **テスト名**: PNG読み込み時のBase64文字列がSTANDARD方式で正しくデコードできる
  - **何をテストするか**: 返却されたbase64フィールドを`base64::STANDARD.decode()`でデコードし、有効なバイナリデータが得られること
  - **期待される動作**: Base64文字列をデコードした結果がPNG画像バイナリとして妥当であること
- **入力値**: テスト用PNGファイルのパス（`tests/fixtures/sample.png`）
  - **入力データの意味**: Base64エンコード→デコードの往復検証
- **期待される結果**:
  - `base64::engine::general_purpose::STANDARD.decode(&result.base64)` がOkを返す
  - デコード結果のバイト列が空でない
  - **期待結果の理由**: フロントエンドでdata URIとして画像を表示するために、有効なBase64データが必須
- **テストの目的**: Base64エンコードの正当性検証
  - **確認ポイント**: STANDARD方式（パディング付き）でエンコードされていること
- 🔵 実装コード（`base64::engine::general_purpose::STANDARD.encode`）から直接導出

### TC-004: 画像寸法が実際のファイルと一致すること

- **テスト名**: 返却されるwidth/heightが実画像の寸法と一致する
  - **何をテストするか**: load_imageが返すwidth/heightが、image crateで直接読み込んだ場合の寸法と一致すること
  - **期待される動作**: `img.dimensions()` で取得した値がそのままImageMetadataに設定される
- **入力値**: テスト用PNGファイル（既知のサイズ、例: 100x50px）
  - **入力データの意味**: 寸法が事前に分かっている画像を使用して正確性を検証
- **期待される結果**:
  - `result.width == 100`（テスト画像の実際の幅）
  - `result.height == 50`（テスト画像の実際の高さ）
  - **期待結果の理由**: フロントエンドでCanvas描画する際に正確な寸法が必要
- **テストの目的**: メタデータ抽出の正確性確認
  - **確認ポイント**: image::open() → dimensions() の結果が正しく伝搬されること
- 🔵 実装コード・要件定義書から直接導出

## 2. 異常系テストケース

### TC-005: 存在しないファイルパスを指定した場合

- **テスト名**: 存在しないファイルパスでエラーが返る
  - **エラーケースの概要**: ファイルシステム上に存在しないパスを指定した場合
  - **エラー処理の重要性**: ユーザーが無効なパスを指定した場合やファイルが削除された場合に適切なエラー通知が必要
- **入力値**: `"/nonexistent/path/to/image.png"`
  - **不正な理由**: ファイルが物理的に存在しないパス
  - **実際の発生シナリオ**: ファイル選択後にファイルが移動・削除された場合、または外部メディアが取り外された場合
- **期待される結果**:
  - `Err(String)` が返る
  - エラーメッセージに `"画像の読み込みに失敗しました"` が含まれる
  - **エラーメッセージの内容**: 日本語でユーザーに分かりやすいメッセージ
  - **システムの安全性**: パニックせずにResult::Errで安全に返却される
- **テストの目的**: ファイル不在時のエラーハンドリング確認
  - **品質保証の観点**: アプリがクラッシュせず適切にエラーを返すこと
- 🔵 要件定義書・タスク定義・実装コードのmap_errから直接導出

### TC-006: 非対応画像形式（GIF等）を指定した場合

- **テスト名**: 非対応形式ファイルでエラーが返る
  - **エラーケースの概要**: PNG/JPEG以外の画像形式ファイルを指定した場合
  - **エラー処理の重要性**: タスク完了条件「非対応形式の場合にエラーを返す」を満たすために必須
- **入力値**: テスト用GIFファイルのパス（`tests/fixtures/sample.gif`）、またはBMP/WebP等の非対応形式
  - **不正な理由**: アプリケーションの対応形式はPNGとJPEGのみ
  - **実際の発生シナリオ**: ユーザーがファイル選択ダイアログで非対応形式のファイルを選択した場合
- **期待される結果**:
  - `Err(String)` が返る
  - エラーメッセージが非対応形式であることを示す内容を含む
  - **エラーメッセージの内容**: ユーザーが対応形式を理解できるメッセージ
  - **システムの安全性**: 非対応形式がPNGにフォールバックせず、明示的にエラーを返す
- **テストの目的**: 非対応形式の拒否動作の確認
  - **品質保証の観点**: TASK-0002完了条件「非対応形式の場合にエラーを返す」の検証
- 🔵 タスク完了条件・要件定義書から直接導出

**実装上の注意**: 現在の実装では `_ => "png"` でフォールバックされるため、このテストはRED状態になる。GREEN化のために `_ => return Err("対応していない画像形式です".to_string())` への修正が必要（要件定義書6章に記載済み）。
