## 1. 機能の概要

### 何をする機能か 🔵

**信頼性**: 🔵 *要件定義書のユーザーストーリー・データフロー図より*

フロントエンド（React）とバックエンド（Rust/Tauri v2）のIPC連携を結合し、アプリケーション全体のEnd-to-End動作を完成させる。具体的には以下の一連のフローを統合する：

1. ファイルダイアログによる画像選択 → Rust `load_image` IPC呼び出し → Canvas上への画像表示
2. 2本の水平線ドラッグによるY軸クリップ範囲指定 → リアルタイム拡大プレビュー表示
3. 保存ダイアログによる保存先指定 → Rust `clip_and_save` IPC呼び出し → クリップ画像保存

### どのような問題を解決するか 🔵

**信頼性**: 🔵 *ユーザーストーリー「As a / So that」より*

- ユーザーが画像のY軸方向の不要部分を素早くクリップし、必要な部分だけの画像を作成できるようにする
- ドラッグ操作中にリアルタイムプレビューで確認しながら正確な範囲指定を可能にする

### 想定されるユーザー 🔵

**信頼性**: 🔵 *ユーザーストーリーより*

- 画像のY軸方向トリミングを頻繁に行うユーザー（Windows環境）

### システム内での位置づけ 🔵

**信頼性**: 🔵 *アーキテクチャ設計・データフロー図より*

- プロジェクト最終タスク（TASK-0007）。Phase 1（Rust IPCコマンド: TASK-0001〜0003）とPhase 2（React UIコンポーネント: TASK-0004〜0006）の全成果物を統合する結合タスク
- 依存タスク: TASK-0002（load_image）、TASK-0003（clip_and_save）、TASK-0006（Toolbar・App状態管理）

**参照したEARS要件**: REQ-001, REQ-002, REQ-003, REQ-004

## 2. 入力・出力の仕様

### 入力パラメータ 🔵

**信頼性**: 🔵 *IPCコマンド定義・型定義より*

#### IPC: `load_image`

| パラメータ | 型 | 制約 | 説明 |
|-----------|-----|------|------|
| `path` | `string` | 絶対パス、PNG/JPG/JPEG形式 | 読み込み対象の画像ファイルパス |

#### IPC: `clip_and_save`

| パラメータ | 型 | 制約 | 説明 |
|-----------|-----|------|------|
| `src_path` | `string` | 絶対パス、既存ファイル | 元画像のファイルパス |
| `top_y` | `number`（u32） | 0 <= top_y < bottom_y | クリップ上端Y座標（ピクセル） |
| `bottom_y` | `number`（u32） | top_y < bottom_y <= height | クリップ下端Y座標（ピクセル） |
| `dest_path` | `string` | 絶対パス、書き込み可能 | 保存先ファイルパス |

#### ユーザー入力

| 入力 | 型 | 説明 |
|------|-----|------|
| ファイル選択 | ダイアログ操作 | `@tauri-apps/plugin-dialog` の `open()` |
| ドラッグ操作 | マウスイベント | Canvas上の水平線ドラッグ（Y座標変更） |
| 保存先選択 | ダイアログ操作 | `@tauri-apps/plugin-dialog` の `save()` |

### 出力値 🔵

**信頼性**: 🔵 *IPCコマンド戻り値・型定義より*

#### IPC: `load_image` 戻り値

```typescript
interface ImageMetadata {
  base64: string;          // Base64エンコード済み画像データ
  width: number;           // 画像幅（ピクセル）
  height: number;          // 画像高さ（ピクセル）
  format: "png" | "jpeg";  // 画像形式
}
```

#### IPC: `clip_and_save` 戻り値

- 成功時: `void`（`Ok(())`）
- 失敗時: `string`（`Err(String)` - エラーメッセージ）

#### UI出力

| 出力 | 説明 |
|------|------|
| Canvas画像描画 | Base64画像 + 2本の水平線 + オーバーレイ |
| プレビュー表示 | 選択範囲の拡大プレビュー（PreviewPanel） |
| エラーメッセージ | `div.error-message` にテキスト表示 |
| ローディング状態 | ボタンのdisabled制御 |

### 入出力の関係性 🔵

**信頼性**: 🔵 *データフロー図より*

```
ファイル選択 → load_image(path) → ImageMetadata → Canvas描画
ドラッグ操作 → ClipRegion{topY, bottomY} → PreviewPanel更新
保存操作 → clip_and_save(src_path, top_y, bottom_y, dest_path) → 成功/エラー通知
```

**参照したEARS要件**: REQ-001, REQ-002, REQ-003, REQ-004
**参照した設計文書**: `docs/design/imgx-clip/dataflow.md` - 全データフロー図

## 3. 制約条件

### パフォーマンス要件 🟡

**信頼性**: 🟡 *要件定義の非機能要件から妥当な推測*

- ドラッグ操作中のプレビュー更新は60fps程度を維持する
- Canvas再描画は`requestAnimationFrame`で最適化する

### セキュリティ要件 🟡

**信頼性**: 🟡 *Tauri v2セキュリティモデルから妥当な推測*

- ファイルアクセスはTauri v2 Capability設定で必要最小限に制限
- 設定ファイル: `src-tauri/capabilities/default.json`
- 許可権限: `dialog:allow-open`, `dialog:allow-save`
- IPC通信はTauri v2の型安全なコマンド定義で保護

### 互換性要件 🔵

**信頼性**: 🔵 *要件定義REQ-401, REQ-402より*

- Windows 11上で動作すること（WebView2/Edge使用のためWindows 10 1803以降）
- 対応画像形式: PNG, JPG/JPEG
- Tauri v2 + React 19 + TypeScript

### アーキテクチャ制約 🔵

**信頼性**: 🔵 *アーキテクチャ設計文書より*

- フロントエンド（React）とバックエンド（Rust）はTauri IPC経由で疎結合に連携
- 画像処理はRust側（`image` crate）で実行、UIはWebView（React）で描画
- 状態管理はApp.tsxの`useState`で一元管理
- コンポーネント責務分離:
  - `Toolbar`: ファイルダイアログ操作、IPC呼び出しトリガー
  - `ImageCanvas`: Canvas描画、ドラッグ検出
  - `PreviewPanel`: 拡大プレビュー表示
  - `App`: 全体状態管理、IPC呼び出し実行

### ビルド・配布制約 🟡

**信頼性**: 🟡 *REQ-403から妥当な推測*

- `cargo tauri build`で単体exe生成
- 軽量な単体実行ファイルとして配布可能

**参照したEARS要件**: REQ-401, REQ-402, REQ-403
**参照した設計文書**: `docs/design/imgx-clip/architecture.md` - 非機能要件の実現方法・技術的制約

## 4. 想定される使用例

### 基本的な使用パターン 🔵

**信頼性**: 🔵 *要件定義のユーザーストーリー・受け入れ基準より*

#### パターン1: 画像読み込み→ドラッグ→プレビュー→保存（正常系E2E）

1. ユーザーが「画像を開く」ボタンをクリック
2. ファイルダイアログでPNG/JPG画像を選択
3. Canvas上に画像が表示され、2本の水平線が上端・下端に配置される
4. ユーザーが水平線をドラッグしてY軸クリップ範囲を指定
5. PreviewPanelにリアルタイムで拡大プレビューが表示される
6. ユーザーが「クリップして保存」ボタンをクリック
7. 保存ダイアログで保存先を指定
8. クリップされた画像が保存される

#### パターン2: ドラッグ中のリアルタイムプレビュー

1. 画像が読み込まれた状態で上端の水平線をドラッグ
2. ドラッグ中にPreviewPanelが即座に更新される
3. 下端の水平線も同様にドラッグ可能
4. 両線の位置に応じたクリップ範囲がプレビューに反映される

### エッジケース 🟡

**信頼性**: 🟡 *要件定義の異常系テストケース・データフロー図から妥当な推測*

#### エッジケース1: 非対応形式の画像読み込み

- **入力**: GIF等の非対応形式ファイルを選択
- **期待動作**: エラーメッセージが表示される（Rust側でエラー返却）
- **UI**: `div.error-message`にエラーテキスト表示

#### エッジケース2: ファイルダイアログキャンセル

- **入力**: ファイル選択ダイアログで「キャンセル」を押下
- **期待動作**: 何も起こらない（`path`が`null`のため処理スキップ）

#### エッジケース3: 保存先の書き込み権限なし

- **入力**: 書き込み不可のディレクトリを保存先に指定
- **期待動作**: エラーメッセージが表示される

#### エッジケース4: 画像未読み込み状態での保存操作

- **入力**: 画像が読み込まれていない状態で「クリップして保存」を押下
- **期待動作**: ボタンがdisabled状態のため操作不可

### エラーケース 🟡

**信頼性**: 🟡 *エラーハンドリングフロー図から妥当な推測*

| ケース | トリガー | 期待動作 |
|--------|---------|----------|
| ファイル読み込み失敗 | 存在しないパス | エラーメッセージ表示 |
| 非対応画像形式 | GIF等を選択 | エラーメッセージ表示 |
| 保存失敗 | 書き込み権限なし | エラーメッセージ表示 |
| IPC通信エラー | Rust側例外 | `try/catch`でキャッチ → エラー表示 |

**参照したEARS要件**: REQ-001〜REQ-004の受け入れ基準
**参照した設計文書**: `docs/design/imgx-clip/dataflow.md` - エラーハンドリングフロー

## 5. EARS要件・設計文書との対応関係

### 参照したユーザーストーリー

- **ストーリー1**: 画像の読み込みとクリップ（REQ-001, REQ-002, REQ-003）
- **ストーリー2**: リアルタイムプレビューで確認（REQ-004）

### 参照した機能要件

- **REQ-001**: 画像ファイル読み込み（PNG/JPG対応）
- **REQ-002**: GUI上でY軸方向ドラッグ操作による範囲指定
- **REQ-003**: 指定Y軸範囲でクリップ・同一形式で保存
- **REQ-004**: ドラッグ操作中のリアルタイム拡大プレビュー

### 参照した非機能要件

- **REQ-401**: Tauri v2 + React TypeScript構築
- **REQ-402**: Windows上での動作
- **REQ-403**: 単体exe配布

### 参照したEdgeケース

- 非対応形式の画像読み込み時のエラーメッセージ表示（REQ-001異常系）
- ドラッグ操作の境界クランプ（REQ-002異常系）
- 保存先書き込み権限なし時のエラーメッセージ表示（REQ-003異常系）

### 参照した受け入れ基準

- 正常系: PNG/JPG画像を読み込み表示できる
- 正常系: ドラッグでY軸範囲を選択できる
- 正常系: 切り取り後の画像が正しいサイズ・形式で保存される
- 正常系: ドラッグ中に選択範囲が拡大表示される
- 異常系: 非対応形式の場合エラーメッセージを表示する
- 異常系: 保存先に書き込み権限がない場合エラーメッセージを表示する

- **アーキテクチャ**: `docs/design/imgx-clip/architecture.md` - システム概要、コンポーネント構成、IPCコマンド定義、非機能要件の実現方法
