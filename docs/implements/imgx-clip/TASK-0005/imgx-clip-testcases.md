## 1. 正常系テストケース（基本的な動作）

### TC-001: Canvas要素がレンダリングされる

- **テスト名**: Canvas要素がレンダリングされる
  - **何をテストするか**: PreviewPanelコンポーネントがマウントされた際に、canvas要素がDOMに存在すること
  - **期待される動作**: コンポーネントが正常にレンダリングされ、canvas要素が生成される
- **入力値**: 有効なprops（imageData=Base64文字列, imageWidth=200, imageHeight=300, topY=0, bottomY=300）
  - **入力データの意味**: 画像が読み込まれた状態の標準的なprops
- **期待される結果**: `<canvas>` 要素がDOMに存在する
  - **期待結果の理由**: PreviewPanelはCanvas APIで描画するため、canvas要素が必須
- **テストの目的**: コンポーネントの基本的なレンダリングを確認
  - **確認ポイント**: canvas要素の存在確認
- 🔵 **青信号**: note.mdのPreviewPanel設計・既存ImageCanvas.test.tsxのパターンより

### TC-002: 初期マウント時にCanvas描画が実行される

- **テスト名**: 初期マウント時にdrawImageが呼ばれる
  - **何をテストするか**: 有効なpropsでコンポーネントがマウントされた際に、Canvas 2DコンテキストのdrawImageが呼ばれること
  - **期待される動作**: useEffect内でBase64画像がロードされ、drawImageでソース範囲指定描画が実行される
- **入力値**: imageData=Base64文字列, imageWidth=200, imageHeight=300, topY=50, bottomY=250
  - **入力データの意味**: 選択範囲が設定された状態での初期描画テスト
- **期待される結果**: `ctx.drawImage` が呼ばれる（少なくとも1回）
  - **期待結果の理由**: 有効な画像データとクリップ範囲が渡されれば描画が実行されるべき
- **テストの目的**: Canvas描画ロジックの初期実行を確認
  - **確認ポイント**: drawImageの呼び出し有無
- 🔵 **青信号**: note.mdのCanvas描画フロー・要件定義REQ-004の受け入れ基準より

### TC-003: drawImageに正しいソース範囲パラメータが渡される

- **テスト名**: drawImageに正しいソース範囲パラメータが渡される
  - **何をテストするか**: drawImageの9引数形式で、ソース範囲（sy=topY, sh=bottomY-topY）が正しく指定されること
  - **期待される動作**: `drawImage(image, 0, topY, imageWidth, bottomY-topY, 0, 0, canvasWidth, canvasHeight)` の形式で呼ばれる
- **入力値**: imageWidth=200, imageHeight=300, topY=50, bottomY=250
  - **入力データの意味**: クリップ高さ200px（250-50）のケース。ソース範囲のsy=50, sh=200が正しく指定されることを検証
- **期待される結果**: drawImageの第3引数（sy）が50、第5引数（sh）が200であること
  - **期待結果の理由**: Canvas描画仕様に基づき、sy=topY, sh=(bottomY-topY)でソース範囲を指定する
- **テストの目的**: Canvas描画パラメータの正確性を確認
  - **確認ポイント**: drawImageの各引数値（特にsy, sw, sh）
- 🔵 **青信号**: note.mdのCanvas描画戦略・要件定義の入出力仕様より

### TC-004: topY変更時にCanvasが再描画される

- **テスト名**: topY変更時にCanvasが再描画される
  - **何をテストするか**: topY propsが変更された際に、clearRectとdrawImageが再度呼ばれること
  - **期待される動作**: useEffectの依存配列にtopYが含まれており、変更時にCanvas再描画がトリガーされる
- **入力値**: 初期: topY=50, bottomY=250 → 変更後: topY=100, bottomY=250
  - **入力データの意味**: ユーザーがImageCanvasで上端線をドラッグし、topYが50から100に移動したシナリオ
- **期待される結果**: drawImageが初期描画後に再度呼ばれる（合計2回以上）
  - **期待結果の理由**: topY変更はプレビュー範囲の変更を意味し、再描画が必須
- **テストの目的**: リアルタイム更新（topY変更追従）を確認
  - **確認ポイント**: rerender後のdrawImage呼び出し回数
- 🔵 **青信号**: 要件定義REQ-004の受け入れ基準「ドラッグ範囲変更に追従して拡大表示が更新される」より

### TC-005: bottomY変更時にCanvasが再描画される

- **テスト名**: bottomY変更時にCanvasが再描画される
  - **何をテストするか**: bottomY propsが変更された際に、Canvasが再描画されること
  - **期待される動作**: useEffectの依存配列にbottomYが含まれており、変更時にCanvas再描画がトリガーされる
- **入力値**: 初期: topY=50, bottomY=250 → 変更後: topY=50, bottomY=200
  - **入力データの意味**: ユーザーがImageCanvasで下端線をドラッグし、bottomYが250から200に移動したシナリオ
- **期待される結果**: drawImageが初期描画後に再度呼ばれる
  - **期待結果の理由**: bottomY変更はクリップ高さの変更を意味し、拡大率も変わるため再描画が必須
- **テストの目的**: リアルタイム更新（bottomY変更追従）を確認
  - **確認ポイント**: rerender後のdrawImage呼び出し回数と更新後のソース範囲パラメータ
- 🔵 **青信号**: 要件定義REQ-004の受け入れ基準より

### TC-006: 再描画前にclearRectが呼ばれる

- **テスト名**: 再描画前にclearRectでCanvasがクリアされる
  - **何をテストするか**: Canvas描画の前にclearRectが呼ばれ、前回の描画内容がクリアされること
  - **期待される動作**: drawImageの前にclearRect(0, 0, canvasWidth, canvasHeight)が呼ばれる
- **入力値**: 有効なprops（imageData=Base64文字列, topY=50, bottomY=250）
  - **入力データの意味**: 描画が実行される標準的な状態
- **期待される結果**: clearRectが呼ばれること
  - **期待結果の理由**: Canvas再描画時に前回の描画内容が残ると表示が崩れるため、クリアが必須
- **テストの目的**: Canvas描画の前処理（クリア）を確認
  - **確認ポイント**: clearRectの呼び出し有無
- 🔵 **青信号**: note.mdのCanvas描画仕様「Canvas描画前にclearRectで初期化すること」より

### TC-007: imageData変更時にImageキャッシュが更新される

- **テスト名**: imageData変更時にプレビューが新しい画像で再描画される
  - **何をテストするか**: imageData propsが変更された際に、新しい画像データでCanvas描画が更新されること
  - **期待される動作**: 新しいBase64データがImageオブジェクトにロードされ、drawImageが再実行される
- **入力値**: 初期: imageData="data:image/png;base64,AAA..." → 変更後: imageData="data:image/png;base64,BBB..."
  - **入力データの意味**: ユーザーが別の画像を読み込んだシナリオ（画像切り替え）
- **期待される結果**: drawImageが新しい画像データで再度呼ばれる
  - **期待結果の理由**: imageData変更はuseEffectの依存配列に含まれ、Imageキャッシュ更新と再描画が必須
- **テストの目的**: 画像切り替え時のプレビュー更新を確認

  - **不正な理由**: 画像データが存在しない状態

- **期待される結果**: drawImageが呼ばれない。canvas要素自体は存在する

- **テストの目的**: null安全なコンポーネント動作を確認
  - **品質保証の観点**: 初期状態でのアプリケーション安定性を保証

## 3. 境界値テストケース（最小値、最大値、null等）

### TC-010: クリップ高さが0の場合（topY === bottomY）

- **テスト名**: topYとbottomYが同じ値の場合にエラーが発生しない
  - **境界値の意味**: クリップ高さが0はdrawImageのsh引数が0になる極端なケース
  - **境界値での動作保証**: ゼロ除算やCanvas描画エラーが発生しないことを保証
- **入力値**: topY=100, bottomY=100, imageWidth=200, imageHeight=300
  - **境界値選択の根拠**: bottomY - topY = 0はクリップ高さの最小境界値
  - **実際の使用場面**: ドラッグ開始直後でtopYとbottomYが同一になる瞬間
- **期待される結果**: エラーなく処理される。drawImageが呼ばれないか、安全にスキップされる
  - **境界での正確性**: クリップ高さ0は描画不要な状態として扱われるべき
  - **一貫した動作**: 高さ0では描画スキップ、高さ>0では描画実行
- **テストの目的**: クリップ高さ0の境界条件を確認
  - **堅牢性の確認**: ゼロ除算等の数値エラーが発生しないこと
- 🟡 **黄信号**: 要件定義の使用例4.5「クリップ高さが0の場合」から妥当な推測

### TC-011: クリップ高さが1pxの場合（最小有効クリップ）

## 6. 要件定義との対応関係

### 参照した機能概要

- **セクション1「機能の概要」**: ドラッグ操作中にY軸の選択範囲をリアルタイムで拡大プレビュー表示する → TC-001〜TC-007
- **REQ-004**: システムはドラッグ操作中に選択範囲をリアルタイムで拡大プレビュー表示しなければならない → TC-004, TC-005

### 参照した入力・出力仕様

- **セクション2「入力パラメータ（Props）」**: IPreviewPanelPropsの型定義 → TC-001〜TC-013の入力値設定
- **セクション2「出力（描画結果）」**: Canvas要素に拡大プレビューを描画 → TC-002, TC-003
- **セクション2「入出力の関係性」**: drawImage(image, 0, topY, imageWidth, bottomY-topY, ...) → TC-003

### 参照した制約条件

- **セクション3「Canvas描画制約」**: drawImageの9引数形式・clearRect初期化 → TC-003, TC-006
- **セクション3「プレビューキャンバスサイズ」**: 固定サイズまたは動的計算 → TC-003のcanvasWidth/canvasHeight
- **セクション3「パフォーマンス要件」**: 大規模画像対応 → TC-013

### 参照した使用例

- **セクション4.1「初期表示」**: topY=0, bottomY=imageHeight → TC-012
- **セクション4.2「ドラッグ中のリアルタイム更新」**: topY/bottomY変更時の再描画 → TC-004, TC-005
- **セクション4.4「imageDataがnullの場合」**: 描画スキップ → TC-008
- **セクション4.5「クリップ高さが0の場合」**: エラーなし処理 → TC-010
- **セクション4.6「imageDataの変更」**: 画像切り替え時の再描画 → TC-007
